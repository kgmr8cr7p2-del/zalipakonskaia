# Taskora

Taskora — внутренняя система управления работой команды на Next.js. В приложении есть общая и личные доски, роли и права доступа, несколько исполнителей у задачи, чат между пользователями, автоматические статусы присутствия, отчёты, история действий, файлы, чек-листы, Telegram-уведомления и адаптивный интерфейс.

## Возможности

- Канбан-доска с drag-and-drop перемещением задач между колонками.
- Режимы «Доска», «Список», «Таймлайн» и «Моя работа».
- Таймлайн в формате диаграммы Ганта с периодами 14, 30 и 60 дней.
- Несколько исполнителей у одной задачи.
- Приоритет, дедлайн, нефтебаза, описание и теги задачи.
- Чек-листы с добавлением, выполнением и удалением пунктов.
- Комментарии и файловые вложения.
- Архивирование и полное удаление задач с учётом прав пользователя.
- Общая доска команды и отдельная личная доска каждого пользователя.
- Поиск, фильтры и экспорт задач в Excel.
- Встроенные личные чаты, поиск пользователя по имени, обмен файлами и отметка прочтения.
- Счётчик непрочитанных сообщений в навигации.
- Профиль с должностью, ником, статусом и настройкой области отображения аватара.
- Автоматический статус пользователя: в сети, создаёт задачу, работает с задачей, смотрит отчёты и другие действия.
- Отчёты, глобальная история изменений и архив.
- Управление пользователями, приглашениями и ролями.
- Настройка колонок выбранной доски.
- Светлая и тёмная темы.
- Адаптация под desktop, планшеты и мобильные устройства.
- Telegram-уведомления о задачах, настраиваемые напоминания о дедлайнах и еженедельный отчёт.
- Страница «Что нового» с мемо последних обновлений продукта.

## Быстрый локальный запуск

### 1. Клонирование

```bash
git clone https://github.com/kgmr8cr7p2-del/KanBan-Table.git
cd KanBan-Table
```

### 2. Переменные окружения

Linux/macOS:

```bash
cp .env.example .env
```

Windows PowerShell:

```powershell
Copy-Item .env.example .env
```

Минимально необходимо проверить эти значения в `.env`:

```dotenv
DATABASE_URL="postgresql://kanban:kanban@localhost:5432/team_kanban?schema=public"
APP_URL="http://localhost:3000"
SESSION_SECRET="replace-with-a-long-random-secret"
```

Для локальной разработки можно сгенерировать секрет командой:

```bash
openssl rand -hex 32
```

### 3. PostgreSQL

Локальный `docker-compose.yml` запускает PostgreSQL на порту `5432`:

```bash
docker compose up -d postgres
docker compose ps
```

Если порт `5432` уже занят другой базой, остановите её или измените проброс порта в `docker-compose.yml` и соответствующий порт в `DATABASE_URL`.

### 4. Зависимости, миграции и начальные данные

```bash
npm ci
npm run prisma:migrate
npm run seed
```

`npm run seed` создаёт:

- роли `ADMIN`, `MANAGER`, `EXECUTOR`;
- общую доску `Taskora`;
- колонки «Новые», «В работе», «На проверке», «Готово»;
- нефтебазу «Не указана».

Seed не создаёт пользователей и не устанавливает стандартные пароли.

### 5. Запуск приложения

```bash
npm run dev
```

Откройте [http://localhost:3000](http://localhost:3000).

## Первый пользователь и администратор

Самостоятельная регистрация разрешена адресам из `INVITED_EMAILS` либо доменам из `ALLOWED_EMAIL_DOMAINS`. Новый пользователь без отдельного приглашения получает роль `EXECUTOR`.

Для первого запуска:

1. Добавьте свой адрес в `INVITED_EMAILS` в `.env`.
2. Перезапустите приложение после изменения `.env`.
3. Зарегистрируйтесь через `/register`.
4. Назначьте первому пользователю роль `ADMIN` один раз через локальную БД:

```bash
node -e "const {PrismaClient}=require('@prisma/client');const p=new PrismaClient();(async()=>{const role=await p.role.findUnique({where:{name:'ADMIN'}});await p.user.update({where:{email:'admin@example.com'},data:{roleId:role.id}});console.log('Admin ready')})()"
```

Замените `admin@example.com` на адрес зарегистрированного пользователя. После повторного входа появится раздел «Админ». Дальнейшие приглашения и роли настраиваются через интерфейс администратора.

## Роли и права

| Действие | ADMIN | MANAGER | EXECUTOR |
|---|:---:|:---:|:---:|
| Просматривать общую доску | Да | Да | Да |
| Создавать задачи общей доски | Да | Да | Нет |
| Редактировать любую задачу | Да | Да | Нет |
| Редактировать назначенную задачу | Да | Да | Да |
| Назначать исполнителей | Да | Да | Нет |
| Настраивать колонки общей доски | Да | Нет | Нет |
| Архивировать и удалять задачи общей доски | Да | Нет | Нет |
| Управлять пользователями и приглашениями | Да | Нет | Нет |

На личной доске владелец может создавать задачи, менять колонки и управлять своими задачами независимо от роли.

## Как работать в приложении

### Доски и задачи

1. Выберите общую или личную доску в выпадающем списке над канбаном.
2. Нажмите «Создать задачу».
3. Заполните название, описание, статус, приоритет и срок.
4. На общей доске выберите одного или нескольких исполнителей.
5. При необходимости добавьте начальные пункты чек-листа и комментарий.
6. После создания откройте карточку, чтобы добавлять файлы, комментарии и новые пункты чек-листа.
7. Перетаскивайте карточку между колонками или меняйте статус в редакторе.

Пункты чек-листа удаляются кнопкой удаления рядом с соответствующим пунктом. Удаление файла или комментария доступно с учётом прав пользователя.

### Режимы представления

- **Доска** — обычный канбан по колонкам.
- **Список** — компактный список задач с основными полями.
- **Таймлайн** — календарная шкала с задачами, дедлайнами, исполнителями и текущим днём.
- **Моя работа** — только задачи, где текущий пользователь указан исполнителем.

На мобильном устройстве канбан и таймлайн прокручиваются внутри своего блока, не растягивая всю страницу.

### Чаты

1. Откройте раздел «Чаты».
2. Найдите пользователя по имени или почте.
3. Откройте диалог и отправьте текст или файл.
4. Непрочитанные сообщения отображаются счётчиком в навигации.
5. У отправленного сообщения отображается состояние прочтения.

### Профиль и присутствие

Профиль открывается нажатием на блок пользователя в левом нижнем углу на desktop или через нижнюю навигацию на мобильном устройстве.

В профиле можно:

- изменить имя, должность и ник;
- загрузить аватар;
- изменить масштаб и положение фотографии перед сохранением;
- удалить аватар;
- увидеть текущий автоматически определяемый статус.

Статус присутствия обновляется приложением в зависимости от активности пользователя и времени последнего действия.

### Настройки колонок

В разделе «Настройки» выберите доску из выпадающего списка. После выбора можно добавлять, переименовывать, переставлять и удалять колонки, если у пользователя есть соответствующие права.

### Отчёты, история и архив

- «Отчёты» показывает показатели задач и активности команды.
- «История» хранит изменения задач и действия пользователей.
- «Архив» содержит архивированные задачи.
- Excel-экспорт учитывает выбранную доску и активные фильтры.

## Переменные окружения

| Переменная | Обязательна | Назначение |
|---|:---:|---|
| `DATABASE_URL` | Да | Строка подключения к PostgreSQL |
| `APP_URL` | Да | Публичный или локальный адрес приложения |
| `SESSION_SECRET` | Да | Секрет сессий и авторизации scheduler-запросов |
| `INVITED_EMAILS` | Нет | Разрешённые email через запятую |
| `ALLOWED_EMAIL_DOMAINS` | Нет | Разрешённые домены через запятую |
| `SMTP_HOST` | Нет | SMTP-сервер |
| `SMTP_PORT` | Нет | SMTP-порт, по умолчанию `587` |
| `SMTP_USER` | Нет | SMTP-пользователь |
| `SMTP_PASS` | Нет | SMTP-пароль |
| `SMTP_FROM` | Нет | Адрес отправителя |
| `TELEGRAM_BOT_TOKEN` | Нет | Токен Telegram-бота |
| `TELEGRAM_BOT_USERNAME` | Нет | Имя бота без `@` для ссылки из настроек; если не задано, приложение получает имя через Bot API |
| `TELEGRAM_DEFAULT_CHAT_ID` | Нет | Chat ID для уведомлений по умолчанию |
| `TELEGRAM_WEBHOOK_SECRET` | Для webhook | Секрет проверки запросов Telegram |

Файл `.env` не должен попадать в Git. В репозитории хранится только безопасный шаблон `.env.example`.

## Telegram

Для Telegram-интеграции:

1. Создайте бота через BotFather.
2. Заполните `TELEGRAM_BOT_TOKEN`.
3. Укажите публичный HTTPS-адрес в `APP_URL`.
4. Задайте случайный `TELEGRAM_WEBHOOK_SECRET`.
5. Зарегистрируйте webhook:

```bash
npm run telegram:webhook
```

Webhook обрабатывается маршрутом `POST /api/telegram/webhook`. Бот используется только для уведомлений: создание задач из Telegram и постоянная кнопка Mini App отключены. Личный чат связывается с аккаунтом одноразовой командой `/connect КОД`, которую пользователь получает в настройках; код действует 15 минут.

Проверку включённых напоминаний можно вызвать авторизованным запросом:

```text
POST /api/notifications/deadlines
```

Контейнер `scheduler` в production проверяет напоминания каждые 30 секунд и отправляет еженедельный отчёт по пятницам в 16:40 по московскому времени. Напоминания отправляются в 09:00 по Москве: для общей доски — в общий канал за день до дедлайна, для личной — в личный чат с выбранным интервалом. Повторная отправка блокируется записью в БД.

## CI/CD и деплой из GitHub

Workflow `.github/workflows/ci-cd.yml` запускается:

- при pull request в `main` — только production-сборка;
- при push в `main` — сборка и автоматический деплой;
- вручную через `workflow_dispatch`.

Для автоматического деплоя в GitHub Environment `production` должен быть добавлен секрет:

- `DEPLOY_SSH_KEY` — приватный SSH-ключ доступа к production-серверу.

На production-сервере ожидаются:

- каталог `/opt/team-kanban-board`;
- файл `/opt/team-kanban-board/.env.production`;
- Docker и Docker Compose;
- внешняя Docker-сеть `regionfree-edge` для reverse proxy.

Если сеть ещё не создана:

```bash
docker network create regionfree-edge
```

При push в `main` GitHub Actions:

1. устанавливает зависимости через `npm ci`;
2. выполняет `npm run build`;
3. упаковывает исходники без секретов и загружаемых файлов;
4. передаёт релиз на сервер по SSH;
5. создаёт резервную копию предыдущей версии;
6. собирает новый Docker-образ;
7. перезапускает приложение и scheduler;
8. выполняет `prisma migrate deploy`;
9. проверяет доступность `/login`;
10. регистрирует Telegram webhook, если задан токен.

При ошибке deploy-скрипт автоматически возвращает предыдущий релиз и Docker-образ.

## Диагностика

### Приложение не подключается к PostgreSQL

```bash
docker compose ps
docker compose logs postgres
```

Проверьте порт, логин, пароль, имя базы и `DATABASE_URL`.

### Prisma Client не соответствует схеме

```bash
npm run prisma:generate
```

Если Windows сообщает, что `query_engine-windows.dll.node` занят, остановите запущенный `npm run dev`, затем повторите команду.

### Регистрация запрещена

Проверьте `INVITED_EMAILS` и `ALLOWED_EMAIL_DOMAINS`, затем перезапустите приложение.

### Не загружается файл

Проверьте существование и права записи каталога `uploads`. В Docker проверьте подключение volume `team-kanban-uploads`.

### Не приходят Telegram-уведомления

Проверьте `TELEGRAM_BOT_TOKEN`, chat ID пользователя, `APP_URL`, webhook и логи контейнеров:

```bash
docker compose -f docker-compose.production.yml logs app scheduler
```

## Безопасность

- Никогда не добавляйте `.env` и `.env.production` в Git.
- Используйте уникальный длинный `SESSION_SECRET`.
- Не публикуйте Telegram-токены, SMTP-пароли и SSH-ключи.
- Ограничьте `INVITED_EMAILS` и `ALLOWED_EMAIL_DOMAINS` рабочими адресами.
- Регулярно обновляйте зависимости и PostgreSQL.
- Перед production-деплоем создавайте резервную копию БД и загруженных файлов.
